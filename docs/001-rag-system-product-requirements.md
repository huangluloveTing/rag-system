## 1. 背景与目标

### 1.1 为什么要做这个系统？

随着企业知识资产不断积累（文档、手册、FAQ、代码库等），员工查找信息的效率成为瓶颈。传统关键词搜索无法理解语义，而纯 LLM 又存在幻觉问题。RAG（检索增强生成）系统结合两者优势，让 AI 基于企业知识库准确回答问题。

### 1.2 解决什么用户问题？

| 用户角色 | 痛点 | RAG 解决方案 |
|----------|------|--------------|
| 客服人员 | 需要快速查找产品文档回答客户问题 | 输入问题，秒级获得带引用的答案 |
| 新员工 | 不熟悉公司内部流程和制度 | 自然语言问答，快速上手 |
| 开发人员 | 需要在多个文档中查找 API/规范 | 代码/文档知识库统一检索 |
| 管理层 | 需要快速了解项目/业务信息 | 跨文档信息整合，自动生成摘要 |

### 1.3 成功指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 检索召回率 | ≥85% | 人工标注测试集 |
| 回答准确率 | ≥90% | 用户点赞率 + 人工抽检 |
| 平均响应时间 | <3 秒 | 系统监控 P95 |
| 用户满意度 | ≥4.5/5 | 反馈评分 |
| 知识库覆盖率 | ≥95% | 文档入库率 |

## 3. 功能描述

### 3.1 核心功能

#### F1 - 知识问答
- 用户输入自然语言问题
- 系统检索相关知识片段
- LLM 生成带引用的回答
- 支持追问（多轮对话上下文）

#### F2 - 文档管理
- 支持格式：PDF、Word、Markdown、TXT、HTML
- 批量上传（拖拽/选择文件）
- 文档分类/标签
- 版本管理（更新后保留历史版本）
- 文档预览

#### F3 - 知识库配置
- 创建/编辑/删除知识库
- 分块策略配置（chunk 大小、重叠率）
- Embedding 模型选择
- 检索参数配置（Top-K、相似度阈值）

#### F4 - 检索优化
- 混合检索（向量 + 关键词）
- Rerank 重排序
- 查询改写（同义词扩展、问题分解）
- 检索结果人工标注（点赞/点踩）

#### F5 - 权限管理
- 用户角色：管理员/编辑/查看者
- 文档级权限控制
- 知识库级权限控制
- 检索结果过滤（无权限文档不显示）

#### F6 - 监控与统计
- 检索日志（问题、检索结果、响应时间）
- 用户反馈统计（点赞/点踩率）
- 热门问题排行
- 无结果问题统计（用于补充知识库）

---

### 3.2 边界情况

| 场景 | 处理方式 |
|------|----------|
| 用户上传超大文件（>100MB） | 分片上传或拒绝并提示 |
| 文档格式不支持 | 提示支持的文件格式列表 |
| 检索无结果 | 明确告知"未找到相关信息"，并提供相似问题推荐 |
| 答案置信度低（相似度<0.3） | 标注"答案可能不准确，请核实" |
| 多知识库检索冲突 | 按优先级/时间排序，标注来源 |
| 敏感词/违规内容 | 内容过滤，拒绝回答并记录日志 |
| 并发请求过高 | 限流降级，返回友好提示 |

---

### 3.3 异常流程

| 异常场景 | 系统行为 | 用户提示 |
|----------|----------|----------|
| LLM API 超时/失败 | 重试 3 次后降级 | "服务暂时繁忙，请稍后重试" |
| 向量数据库不可用 | 切换到备用节点/只读模式 | "检索服务维护中，稍后恢复" |
| 文档解析失败 | 跳过该文档，记录错误日志 | "文档 X 解析失败，请检查格式" |
| Embedding 生成失败 | 重试或标记文档待处理 | "文档 X 处理中，稍后完成" |
| 用户无权限访问 | 过滤检索结果 | 不提示（静默过滤） |

## 4. 验收标准

### 4.1 功能验收

- [ ] 用户上传 PDF/Word/Markdown 文档后，可在 5 分钟内完成索引
- [ ] 用户提问后，系统在 3 秒内返回带引用的答案
- [ ] 答案引用可点击跳转到原文位置
- [ ] 管理员可设置文档权限，无权限用户检索不到
- [ ] 系统记录检索日志，可导出 CSV
- [ ] 用户可对答案点赞/点踩，反馈计入统计

### 4.2 性能验收

- [ ] 支持并发 100 QPS 检索请求
- [ ] 向量检索 P95 延迟 <500ms
- [ ] 支持 100 万 + 文档片段索引
- [ ] 文档上传后索引构建时间 <10 分钟/万页

### 4.3 质量验收

- [ ] 检索召回率 ≥85%（测试集评估）
- [ ] 回答准确率 ≥90%（人工抽检）
- [ ] 单元测试覆盖率 ≥80%
- [ ] 无 P0/P1 级别 Bug

## 5. 优先级（MoSCoW）

### Must have（必须有）
- 文档上传与解析（PDF/Word/Markdown/TXT）
- 向量化与索引构建
- 语义检索（Top-K 相似度搜索）
- LLM 生成带引用的回答
- 基础用户认证与权限
- 检索日志记录

### Should have（应该有）
- 混合检索（向量 + 关键词）
- Rerank 重排序
- 文档分类与标签
- 用户反馈（点赞/点踩）
- 监控仪表盘（检索量、满意度）
- 多知识库支持

### Could have（可以有）
- 多轮对话（上下文记忆）
- 文件直接提问（上传即问）
- 查询改写（同义词扩展）
- 答案置信度提示
- 热门问题推荐
- API 开放接口

### Won't have（本次不做）
- 语音问答
- 多模态检索（图片/视频内容）
- 自动知识库更新（爬虫）
- 多语言自动翻译
- 移动端 App

## 6. 风险与待确认

### 6.1 技术风险

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| Embedding 模型中文效果差 | 检索准确率低 | 预测试多种模型（bge/m3e/text2vec） |
| LLM 幻觉严重 | 答案不可信 | 强制引用标注 + 置信度提示 |
| 向量数据库性能瓶颈 | 响应慢 | 压力测试 + HNSW 索引优化 |
| 敏感信息泄露 | 合规风险 | 权限过滤 + 内容审计 + 日志脱敏 |

### 6.2 待确认事项

- [ ] **知识库规模**：预计初期多少文档？（影响技术选型）
- [ ] **并发量**：预计日活用户数？峰值 QPS？
- [ ] **部署方式**：云端 API 还是私有化部署？
- [ ] **预算范围**：LLM Token 成本、向量数据库成本
- [ ] **集成需求**：是否需要与现有系统（OA/IM）集成？
- [ ] **合规要求**：是否有数据出境/审计要求？

### 6.3 依赖项

- LLM API 账号（通义千问/GLM/自建）
- 向量数据库（Milvus/Qdrant/Chroma）
- 文档解析服务（PDF/Office 解析库）
- 用户认证系统（可复用现有或新建）

## 7. 附录

### 7.1 常见 RAG 系统功能对比

| 功能模块 | 开源方案 | 商业方案 |
|----------|----------|----------|
| **文档解析** | LangChain、LlamaIndex | Dify、FastGPT |
| **向量数据库** | Chroma、Milvus、Qdrant | Pinecone、Weaviate Cloud |
| **Embedding** | bge-large-zh、m3e-base | OpenAI text-embedding-3 |
| **Rerank** | bge-reranker、Cohere | 通义千问 Rerank API |
| **LLM** | Qwen、GLM、ChatGLM | GPT-4、Claude、通义千问 |
| **完整平台** | Dify、LangChain-Chatchat | 阿里百炼、百度千帆、FastGPT |

### 7.2 推荐技术栈（MVP）

```
前端：React + Ant Design
后端：Python FastAPI / Node.js NestJS
向量库：Chroma（MVP）→ Milvus（生产）
Embedding: bge-large-zh-v1.5
Rerank: bge-reranker-large
LLM: 通义千问 Qwen3.5-plus
文档解析：PyMuPDF（PDF）、python-docx（Word）
```
